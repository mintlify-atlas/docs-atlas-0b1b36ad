## Description

The `workspace` tool manages shadow workspace transactions, providing atomic multi-file operations with rollback support, architectural auditing, and dependency tracking.

## Request Parameters

<ParamField path="action" type="string" required>
  Workspace operation:
  - `status` - Get workspace and transaction status
  - `list` - List all files in workspace
  - `dependencies` - Detect and list package dependencies
  - `begin` - Start a new transaction session
  - `diff` - Show uncommitted changes
  - `commit` - Atomically write changes to disk
  - `rollback` - Discard changes and end session
  - `alerts` - View architectural alerts and fault reports
  - `clear_faults` - Clear all semantic faults
</ParamField>

<ParamField path="profile" type="string">
  Execution profile for capability checks
</ParamField>

## Response Fields

Responses vary by action. See action-specific schemas below.

## Actions

### status

Get current workspace and transaction state.

#### Response

<ResponseField name="transaction_active" type="boolean" required>
  Whether a shadow session is active
</ResponseField>

<ResponseField name="staged_files" type="array" required>
  Relative paths of files with uncommitted changes
</ResponseField>

<ResponseField name="staged_count" type="number" required>
  Number of staged files
</ResponseField>

<ResponseField name="files_found" type="number" required>
  Total source files in workspace
</ResponseField>

<ResponseField name="state_hash" type="string">
  Current workspace state hash (for optimistic concurrency control)
</ResponseField>

#### Example

```json Request
{"action": "status"}
```

```json Response
{
  "transaction_active": true,
  "staged_files": ["src/lib.rs", "src/utils.rs"],
  "staged_count": 2,
  "files_found": 47,
  "state_hash": "a3f5e8c9d4b2a1f6e8c7d5b3a2f1e9c8"
}
```

## Transaction Workflow

### Typical Workflow

```json
// 1. Start session
{"tool": "workspace", "action": "begin"}

// 2. Make changes
{"tool": "edit", "uri": "src/lib.rs::foo", "code": "...", "action": "upsert"}
{"tool": "edit", "uri": "src/lib.rs::bar", "code": "...", "action": "upsert"}

// 3. Review changes
{"tool": "workspace", "action": "diff"}

// 4. Commit
{"tool": "workspace", "action": "commit"}

// 5. Check alerts
{"tool": "workspace", "action": "alerts"}
```

### Rollback on Error

```json
// 1. Start session
{"tool": "workspace", "action": "begin"}

// 2. Make changes
{"tool": "edit", "uri": "src/lib.rs::foo", "code": "...", "action": "upsert"}

// 3. Oops, mistake detected!
{"tool": "workspace", "action": "rollback"}

// 4. Start fresh
{"tool": "workspace", "action": "begin"}
```

## Shadow Workspace

The shadow workspace provides **physical isolation** for uncommitted changes:

- **Location**: `.curd/shadow/root/`
- **Creation**: Cloned on `begin`
- **Isolation**: All edits, reads, and builds use shadow files
- **Atomicity**: Commit uses three-way merge for conflict resolution

### Conflict Resolution

If base files changed during transaction:

1. **Three-way merge** using `git merge-file`
2. Conflict markers inserted if merge fails
3. Commit succeeds with warnings

```json Commit with conflicts
{
  "status": "ok",
  "committed": ["src/lib.rs"],
  "files_written": 1,
  "conflicts": [
    {
      "file": "src/lib.rs",
      "message": "Merge conflict detected. Review conflict markers."
    }
  ]
}
```

## Nested Transactions (Savepoints)

CURD supports **nested savepoints** for incremental commits:

```json
// 1. Begin outer transaction
{"tool": "workspace", "action": "begin"}

// 2. Make changes
{"tool": "edit", "uri": "src/lib.rs::foo", "code": "..."}

// 3. Create savepoint (commit inner transaction)
{"tool": "workspace", "action": "commit"}
// Response: {"message": "Committed nested savepoint."}

// 4. Make more changes
{"tool": "edit", "uri": "src/lib.rs::bar", "code": "..."}

// 5. Rollback to savepoint
{"tool": "workspace", "action": "rollback"}
// Response: {"message": "Rolled back to previous savepoint. Current staged files: 1"}

// 6. Commit outer transaction
{"tool": "workspace", "action": "commit"}
```

## Architectural Auditing

Post-commit audits detect:

### Cycle Detection

```json
{
  "cycles": [
    ["src/a.rs::foo", "src/b.rs::bar", "src/a.rs::foo"]
  ]
}
```

### Cohesion Degradation

```json
{
  "semantic_violations": [
    {
      "symbol": "src/lib.rs::calculate",
      "severity": "high",
      "message": "Cohesion ratio dropped: 0.65 -> 0.45"
    }
  ]
}
```

### Broken Imports

```json
{
  "broken_imports": [
    {
      "file": "src/lib.rs",
      "line": 10,
      "import": "use crate::missing::Foo;",
      "message": "Symbol 'Foo' not found"
    }
  ]
}
```

## Dependency Detection

Supported package managers:

| Package Manager | Manifest File | Languages |
|-----------------|---------------|------------|
| Cargo | `Cargo.toml` | Rust |
| npm/yarn/pnpm | `package.json` | JavaScript/TypeScript |
| pip | `requirements.txt`, `pyproject.toml` | Python |
| go | `go.mod` | Go |
| bazel | `BUILD.bazel` | Multi-language |
| cmake | `CMakeLists.txt` | C/C++ |
| vcpkg | `vcpkg.json` | C/C++ |

## Error Handling

```json Error: No active transaction
{
  "error": "No active transaction. Call 'begin' first."
}
```

```json Error: Already in transaction
{
  "error": "Transaction already active. Commit or rollback first."
}
```

```json Error: Commit failed
{
  "error": "Commit failed: merge conflict in src/lib.rs"
}
```
